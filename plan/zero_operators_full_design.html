plan.md= your SoW encoded
ivl-f5-soft-sensor.md is your plan.md. It contains: objectives, oracle definition (RMSE targets, tier thresholds), data sources with schema, exclusion rules (TA dates, _MA tags, AC2658), agent instructions, human checkpoint positions, and out-of-scope boundaries. Agents read this at session start. You edit it when scope changes.
SKILL.md= encoded expertise
The ml-workflow skill teaches every agent how ML jobs are structured — the full pipeline from data review through inference packaging. The logging skill defines how agents write to the comms log. The pytorch-patterns skill gives the ML engineer your preferences (DataLoader patterns, GPU optimisation, experiment tracking with wandb/mlflow).
The research team — 7 agents, each a .md file in .claude/agents/
Lead orchestratorResearch director
Reads memory + plan.md at session start. Decides mode (build/continue/maintain). Decomposes plan into phases. Issues contracts to subagents — each contract specifies: inputs, expected outputs, acceptance criteria, time budget. Gates parallel spawning behind sequential groundwork. Writes STATE.md and DECISION_LOG.md. Never writes code itself.
Data engineerPipeline + data integrity
Owns everything before the model sees data. Reads IP.21 + SAP QM exports. Applies all filters (TA exclusions, _MA=manual, AC2658 bypass periods). Nearest-timestamp join (±15min). Writes quality report: missing data %, tag coverage, anomaly flags. Defines the PyTorch DataLoader. Output: clean data/ directory + quality_report.md. Blocks model build until quality gates pass.
Time series researcherModel architecture + training
Your ML scientist. Selects model architecture (gradient boosting, LSTM, transformer — justified against data characteristics). Trains soft sensor against F5LB ground truth. Handles pre/post-TA regime segmentation. Runs overnight iteration loop against oracle. Logs every experiment to logs/experiments/ with full hyperparams. Heavy PyTorch user — uses your DataLoader patterns. Reports to lead after each iteration batch.
ML engineerSpeed, infra, packaging
Makes it run fast and reliably. GPU/multi-GPU optimisation, mixed precision, torch.compile where appropriate. Experiment tracking (wandb or mlflow). Writes the inference script. Packages the final model artefact. Handles the 6-hour recalibration loop logic. Peer-communicates with TS researcher on architecture decisions that affect training speed.
Domain evaluatorPetrochemical process validity
The most important agent nobody else builds. Validates that model outputs make physical sense. Does the predicted PO purity trend correctly when EPOX reactor temp increases? Are TBHP breakthrough predictions physically consistent with known epoxidation kinetics? Flags when a statistically good model produces process-nonsensical outputs. Reads the domain glossary and SoW. Peer-challenges the TS researcher's feature selection decisions.
Oracle / QA agentTier testing + RMSE gating
Runs the hard oracle automatically. Computes RMSE per output tag against held-out lab samples. Runs Tier 1/2/3 question sets from tier_questions.py. Reports structured pass/fail with per-question breakdown. Blocks proceed signal if below threshold. Peer-messages lead orchestrator with results. Also runs drift detection oracle — does the drift alerting logic trigger correctly on known drift events from the Nov 2025 TA period?
Infra / integration engineerEnvironment + deployment
Owns the runtime environment. Sets up the Python environment (uv, pyproject.toml). Manages the dashboard web server. Writes the logging infrastructure that feeds the comms wall. Handles data pipeline scheduling (6hr recalibration). For later: manages the Orbital integration layer. Peer-communicates with ML engineer on GPU environment setup.
The ML workflow — what the ml-workflow SKILL.md encodes
1
Phase 1: Data review + validation — sequential, data engineer solo
Plot and understand: distribution of each output tag, time series plots for all 4.5yr history, identify regime changes visually. General stats: mean/std/skew per tag, missing data heatmap, correlation matrix vs outputs. Data hygiene report: duplicate timestamps, sensor saturation events, physically impossible values.

Subtasks (sequential within this phase):
1a. Raw data audit → quality_report.md
1b. Exclusion filter application (TA, _MA, AC2658)
1c. Nearest-timestamp join + alignment validation
1d. Define PyTorch DataLoader + dataset class
1e. Define train/val/test split function (time-aware — no leakage)
GATE
Lead orchestrator reviews quality_report.md. Human checkpoint: you + IVL engineer review feature list. Approve → Phase 2 spawns. Reject → data engineer iterates.
2
Phase 2: Feature selection — sequential, TS researcher + domain evaluator pair
15,601 → 200-400 tags. Correlation filter vs each output tag. VIF/multicollinearity pruning. Control loop completeness (PV + SP + OP + _MA). Domain evaluator validates physical plausibility of selected features. Ranked output: feature list with correlation, VIF, physical justification per tag.
GATE
Human checkpoint — you review with IVL process engineer. This is the one mandatory human gate. Approve → Phase 3 spawns all parallel agents.
3
Phase 3: Model training + evaluation — parallel spawn
This is where parallelism begins. Multiple model architectures explored simultaneously.
PARALLEL — spawned simultaneously
TS researcher: gradient boosting baseline
TS researcher: LSTM experiment
ML engineer: GPU setup + experiment tracking
Oracle agent: drift detection test suite
4
Phase 4: Benchmarking + iteration loop — overnight autonomous, autoresearch pattern
Oracle agent scores each iteration. RMSE per output tag. Tier 1/2 question pass rates. TS researcher iterates on best-performing architecture. Domain evaluator challenges each iteration for physical validity. Loop continues autonomously until oracle threshold met or iteration budget exhausted. You wake up to: iteration log, best model, scores, domain evaluator notes.
GATE
Oracle gate: Tier 1 ≥ 80%, RMSE within target. Automatic proceed if passed. Human review if failed — you decide: iterate more, change architecture, revisit feature selection.
5
Phase 5: Model packaging + Tier 3 — parallel spawn
Inference script. Recalibration loop (6hr cadence logic). Drift detection integration. Model card (performance by regime, by output tag, confidence intervals). QA agent runs full Tier 1/2/3 question set. Domain evaluator writes process validity sign-off. All feeds into validation_report.md.
PARALLEL — spawned simultaneously
ML engineer: inference script + packaging
Oracle/QA agent: full tier question run
Domain evaluator: process validity report
Infra engineer: Orbital integration prep
The comms wall — what the dashboard sub-app shows
Every agent message is written to logs/comms/session-{date}.jsonl in a structured format. The dashboard renders it as a social-media-style wall — threaded by topic, filterable by agent pair, timestamped. This is the peer message feed that no other tool surfaces.

JSONL log schema — what each message writes
{"ts":"2026-04-08T14:08:22","from":"oracle-agent","to":"lead-orchestrator", "type":"oracle_result","phase":4,"iteration":7, "content":"Tier 1: 78%...", "metrics":{"rmse_avg":0.84,"tier1":0.78,"tier2":0.71}, "action_required":true,"checkpoint":false}
Dashboard reads the JSONL live, groups by thread (from+to pair), renders newest first, filterable by agent, phase, and action_required flag. The "action_required" flag highlights messages in amber. "checkpoint" flag triggers a push notification to you. Future: add voting/approve buttons directly in the wall so you can respond from the dashboard without touching the terminal.
