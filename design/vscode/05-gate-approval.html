How much autonomy

Gate mode is a dial, not a rewrite. Flip it mid-run and the orchestrator picks up the new setting at the next phase boundary.

GATE-MODE CONTROLSUPERVISED ACTIVE
supervised Every gate blocks until a human approves, rejects, or iterates.
auto Gates self-approve when the oracle clears must_pass; anything below escalates to you.
full-auto No blocking gates. Oracle results are recorded and the run continues to packaging.
writes <memory>/gate_mode via zo gates set — hot-reloaded by the running orchestrator

Implementation notes

  • Card data is a GateEvent (gate_name, metric, value, tier thresholds, per-class breakdown) tailed from logs/comms/2026-07-12.jsonl via a workspace FileSystemWatcher; no tmux polling.
  • Approve executes zo runtime send --pane %12 --text "/gates:approve" --submit; the lead pane id is resolved from .zo/runtime/session.json, raw output verifiable in panes/p12.log.
  • Reject & rework injects /gates:reject "<reason>"; the lead records the reason to DECISION_LOG.md before rework. Iterate maps to /gates:reject with iterate intent (same pane, reason auto-prefixed).
  • The card clears only when a gate_resolved event appears in the comms tail; buttons show an in-flight state until then, with a timeout warning if the pane never echoes. No optimistic phase advance in v1.
  • Gate-mode segmented control shells out to zo gates set <mode>, which writes <memory>/gate_mode; the running orchestrator hot-reloads it at the next phase boundary, no session restart.
  • Toast uses vscode.window.showInformationMessage with Open / Later actions; Open reveals the ZO sidebar and focuses the gate card. Later snoozes, the status-bar item stays coral.
  • Status-bar fragment is a vscode.window.createStatusBarItem (left-aligned, high priority) that turns coral only while a gate is pending; click performs the same Open action.
  • Per-class breakdown caps at 12 rows in v1; richer confusion drill-down defers to the report webview page rather than growing this card.
