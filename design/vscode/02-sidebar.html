Implementation notes

  • One native TreeDataProvider in the ZO view container; the four states are the same provider swapping roots. State is derived, never stored: no .zo/ → A; .zo/runtime/session.json heartbeat < 90 s → C; heartbeat stale but tmux has-session succeeds → D; otherwise B.
  • Idle phase rows parse the phases table in STATE.md; the plan row opens .zo/plans/mnist-digit-classifier.md via vscode.open; experiment rows enumerate .zo/experiments/*/ and read tier (must/should/could) from each oracle result.
  • Running team rows tail logs/comms/2026-07-12.jsonl for agent status events; the training lineage and 2px epoch rule poll .zo/experiments/exp-003/metrics.jsonl + training_status.json on a 2 s fs.watch debounce (epoch/total → fill width).
  • GATES node reads gate mode from plan frontmatter. The row is read-only in v1 — selecting it reveals the gate approval card; /gates:approve · /gates:reject remain terminal actions until PR D3.
  • Buttons shell out to the CLI in the ZO terminal: Connect a project → zo init <project> / zo draft; Resume → zo continue --headless-tmux; Reattach re-binds panes from .zo/runtime/session.json (e.g. panes/p12.log) without restarting agents.
  • Row grammar: 22 px rows, 16 px per indent level, 11 px JetBrains Mono labels, right-aligned tags; hover uses --surface-2, which maps to list.hoverBackground under the "inherit editor theme" setting.
  • Color discipline: coral marks live activity only (running dots, epoch rule, NEXT); dusk = pending/attention; moss = passed/alive; everything else stays in the text ramp.
  • Deferred from v1: multi-project workspaces (one root per window), drag-to-compare experiments, and inline gate actions in the tree.
