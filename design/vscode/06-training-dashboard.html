Implementation notes

  • Data source: .zo/experiments/exp-003/metrics.jsonl (one JSON object per epoch) + .zo/experiments/exp-003/training_status.json (epoch, eta, device, state), resolved via resolve_active_experiment_dir from .zo/runtime/session.json. No CLI process is spawned — the webview reads what the training loop already writes.
  • Re-render trigger: vscode.workspace.createFileSystemWatcher on both files, debounced ~250 ms; the extension host posts the parsed payload to the webview via postMessage. No polling.
  • Chart is hand-rolled inline SVG — no chart library (webview CSP forbids CDN scripts, and a bundled lib costs more than ~80 lines of path math). Curves recompute from the full metrics.jsonl on each message; hatched region = epochs not yet reached. train_loss strokes var(--text-2) (cream-2 in dark) so it stays legible when the panel flips to the light theme.
  • Stat tiles = last metrics.jsonl line; best val_acc + epoch come from the running max in training_status.json. Progress rule is epoch/epochs from the same file.
  • Checkpoints table lists .zo/experiments/exp-003/checkpoints/ via workspace.fs.readDirectory; kind/cadence from the plan's should_checkpoint: every, best, last. The resumable note reflects the saved state dict (model + optim + sched + RNG) that zo continue restores.
  • Experiment picker enumerates .zo/experiments/*/training_status.json; Δ parent = child best val_acc minus the parent experiment's best. Selecting a row retargets the FileSystemWatcher — completed runs render the same dashboard, static.
  • Terminal panel is the real tmux pane (tail of panes/p12.log) surfaced through the existing terminal integration — the webview renders metrics only; it never re-implements the terminal. Status-bar items via vscode.window.createStatusBarItem, fed from .zo/runtime/session.json.
  • Deferred from v1: per-step (intra-epoch) loss points, ETA regression curve, and two-experiment overlay comparison on one chart.
