Implementation notes

  • v1 input model is deliberate: read-only streams plus the composer. No per-keystroke TTY passthrough, which avoids the entire TTY fragility class (raw mode, resize races, escape-sequence leakage). Composer submits via zo runtime send --pane %12 --submit.
  • Pane roster, tmux pane ids, and cols×rows come from the runtime contract file .zo/runtime/session.json; status dots (working / training / watching / idle) hydrate from agent heartbeat events in logs/comms/2026-07-12.jsonl.
  • Stream transport: tmux pipe-pane mirrors each pane to .zo/runtime/panes/p12.log; the extension host tails the file and feeds xterm.js via Terminal.write(). No onData wiring back to the PTY.
  • Each xterm.js instance is initialized at the manifest cols×rows. Container resizes are debounced 250 ms, then reconciled through zo runtime resize so tmux, the PTY, and the renderer never disagree.
  • Reconnect (window reload or tmux reattach) replays the pane log from the last acknowledged byte offset before switching to live tail; the dusk banner reports replay origin and size, then dismisses itself.
  • Pane death is detected by a tmux pane-died hook updating .zo/runtime/session.json; the overlay keeps the final frame at 40% and "Open log" opens the preserved log with vscode.window.showTextDocument.
  • The grid toggle re-parents the live xterm instances into a 1px-gap CSS grid (no re-spawn); the chosen layout persists in workspaceState. 2×2 is the v1 maximum; free-form splits and pane maximize are deferred.
  • Surface: a WebviewView contributed to the panel area (zeroOperators.terminal); scrollback cap (10k lines), copy-on-select, and find come from stock xterm.js addons.
