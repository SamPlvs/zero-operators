Four regions, four sources.

Every region of the workbench is a thin view over a file the ZO runtime already writes. Nothing here invents state — the extension watches, tails and renders.

Implementation notes

  • Sidebar tree is a vscode.TreeDataProvider in a dedicated view container; it fires onDidChangeTreeData from an fs watcher on .zo/runtime/session.json — no polling loop.
  • Agent editor tabs are webview-hosted xterm.js instances replaying .zo/runtime/panes/p12.log (tmux pipe-pane); keystrokes route through zo runtime send so the extension never drives tmux directly.
  • Phases breadcrumb parses the phase table from STATE.md on activation and on save; the active phase also feeds the status-bar zo: phase_4 item.
  • COMMS panel does an incremental tail of logs/comms/2026-07-12.jsonl; kind → accent mapping is checkpoint dim / message default / decision dusk / gate moss. Clicking a row reveals the source JSONL line; filter chips deferred to v1.1.
  • Status bar is four StatusBarItem entries fed by the training_status.json heartbeat, debounced 2s; session: live flips to a dim detached when the heartbeat lapses beyond 15s.
  • Gate actions (approve / reject from the workbench) invoke /gates:approve and /gates:reject via CLI passthrough — the full gate flow is mocked on its own page, not here.
  • Session bootstrap: the extension attaches to a run started by zo build or zo continue --headless-tmux; it never spawns the team itself in v1.
  • Theming: chrome uses semantic vars mapped to --vscode-* fallbacks in production; terminals intentionally stay dark in light theme, matching VS Code behaviour.
