Implementation notes

  • Source: the extension host tails logs/comms/2026-07-12.jsonl at a persisted byte offset (fs.watch + incremental read) and streams parsed events to this webview via postMessage — the webview itself never polls or touches disk.
  • Daily rollover: at local midnight the tailer closes the old file, opens logs/comms/{next-date}.jsonl at offset 0, and the feed inserts a dim date-divider row.
  • Dedup key is (file, byte offset), not event_id — a retried agent may re-emit an event with the same id, but a line's offset is unique per file.
  • Every event carries a project field; the tailer drops non-matching projects so concurrent ZO sessions on one machine never cross-pollute a feed.
  • Filter chips and search run client-side over an in-memory ring buffer (default 500 events); unknown event types render as dim raw-JSON rows for forward compatibility. Older days stay in JSONL, readable via zo logs.
  • Auto-scroll pin: pinned follows the tail; unpinned freezes the viewport and accrues the coral new-events pill — clicking the pill scrolls to tail and re-pins.
  • Gate cards act: the evaluating card will expose approve/reject that run zo runtime send with /gates:approve · /gates:reject into the lead pane; PASS cards deep-link to the gate record in STATE.md.
  • Deferred from v1: cross-day pagination and full-text search over historical days — the feed shows the current day's tail only.
