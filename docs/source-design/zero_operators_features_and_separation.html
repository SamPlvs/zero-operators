How ZO operates on IVL F5 from outside
The key is --cwd — Claude Code's working directory flag. Zero Operators launches agents pointed at the IVL F5 repo. Agents read and write there. ZO's own state lives in the ZO repo. Two separate git repos, one operation.
ZO orchestratorreadszero-operators/memory/STATE.md
ZO orchestratorspawns agent with--cwd ../ivl_f5
data engineer agentwrites toivl_f5/data/processed/
TS researcher agentwrites toivl_f5/src/models/ + experiments/
ZO orchestratorwrites session summary tozero-operators/memory/sessions/
ZO logswritten tozero-operators/logs/comms/— never touches ivl_f5
target file — how ZO knows where to operate
## Target: IVL F5 Soft Sensor target_repo: ../ivl_f5 target_branch: main worktree_base: ../ivl_f5_worktrees/ ## Agent working directories data_engineer_cwd: ../ivl_f5 ts_researcher_cwd: ../ivl_f5_worktrees/xgboost-arm tft_researcher_cwd: ../ivl_f5_worktrees/tft-arm ml_engineer_cwd: ../ivl_f5 oracle_cwd: ../ivl_f5 ## Output boundaries (ZO never writes these to ivl_f5) zo_only_paths: - zero-operators/memory/ - zero-operators/logs/ - zero-operators/dashboard/
The target file is the contract between ZO and the patient repo. ZO reads it to know where to point agents. The zo_only_paths list is enforced — agents that try to write ZO internals into ivl_f5 are blocked by the orchestrator.
THE BLIND TEST
Your team is given access to the ivl_f5 repo. They review the code, the model, the validation report. They don't know Zero Operators ran it. The test: does the output look like a senior ML engineering team delivered it? If yes — Zero Operators works. If they can tell an AI ran it — you iterate on the agent quality until they can't.
What your team sees in ivl_f5
Clean Python. Typed functions with docstrings. Logical module structure (data/, models/, evaluation/). Pytest test suite. Git history with meaningful commit messages ([data-engineer] feat: add TA exclusion filter for Sep-Nov 2025 period). A validation_report.md that reads like a professional technical document. SHAP outputs for every prediction. A model that passes Tier 1 at 80%+. No mention of Claude, agents, or Zero Operators anywhere.
What they DON'T see
No CLAUDE.md in ivl_f5. No .claude/ directory in ivl_f5. No STATE.md or DECISION_LOG.md. No agent spawn prompts. No ZO logs. No references to orchestration. The git author on commits can be set to "AC Engineering" or individual team member names if you want full deniability. The commits from --cwd ../ivl_f5 agents appear as normal commits in ivl_f5's git history.
The git author trick
# In zero-operators/plans/ivl-f5.md, specify commit identity: git_author_name: "AC ML Engineering" git_author_email: "ml@appliedcomputing.ai" # Agents use this when committing to ivl_f5: GIT_AUTHOR_NAME="AC ML Engineering" \ GIT_AUTHOR_EMAIL="ml@appliedcomputing.ai" \ git commit -m "feat: add XGBoost soft sensor with SHAP explanations"
Or you set this per-agent in their spawn prompt: "When committing, use author 'AC ML Engineering'. Never mention Claude, agents, or Zero Operators in commit messages or code comments."
Metrics for the blind test
Ask your team: "Review this repo and rate it on: code quality (1-10), documentation (1-10), model validity (1-10), would you trust this in production (yes/no)." Then reveal Zero Operators delivered it. Track the gap between how they'd rate human work vs ZO work. That gap is your iteration target.
Why this is the right first test
The blind test proves Zero Operators' core value proposition better than any benchmark: it's not faster AI assistance — it's a digital engineering team that produces output indistinguishable from a human team. If that's true, the commercial implication for Applied Computing is significant.
