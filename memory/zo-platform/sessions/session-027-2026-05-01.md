# Session 027 — STATE.md status validation hardening

**Date:** 2026-05-01
**Duration:** short
**Outcome:** PR #64 opened — `parse_state` validates phase statuses at STATE.md load time so `zo continue` fails fast with a clear, actionable error instead of crashing in `_restore_phase_states` with no hint at the offending file.

## Summary

A real `zo continue --repo .` run on prod-001 crashed with:

```
File "/home/sam/zero-operators/src/zo/orchestrator.py", line 318, in _restore_phase_states
    phase.status = PhaseStatus(saved_states[phase.phase_id])
ValueError: 'prep_complete' is not a valid PhaseStatus
```

The stack trace pointed at `enum.py` and the orchestrator. Nothing said "STATE.md", nothing named the offending value, nothing listed the valid options. Root cause: a hand-edited STATE.md with `phase_3: prep_complete` (operator inventing a status to express "prep wave is done; main wave next") and `phase_3_main: pending` (operator inventing a phase_id). The `phase_3_main:` line was silently dropped by `_PHASE_LINE_RE` (matches `phase_<digits>:` only), but `phase_3: prep_complete` matched and `prep_complete` flowed into `phase_states` as a raw string. The orchestrator then coerced it via `PhaseStatus(value)` and crashed.

## What changed

`src/zo/_memory_formats.py` (+12 lines):
- New module-level `_VALID_PHASE_STATUSES: frozenset[str]` mirroring `PhaseStatus`'s six values, with a comment pointing back to the enum.
- In `parse_state`, after extracting `(pid, status, rest)` from a `## Phases` line, validate `status` against the frozenset and raise `ValueError` with a message naming the offending phase, status, file, and valid options.

`tests/unit/test_memory.py` (+62 lines):
- New `TestPhaseStatusValidation` class with three tests: drift guard (`{s.value for s in PhaseStatus} == _VALID_PHASE_STATUSES`), unknown-status rejection (assertions on the error message contents), and all-six-valid-statuses round-trip.

`memory/zo-platform/STATE.md`: Completed bullet, last_session, last_checkpoint, test_count, prs all cascaded.
`memory/zo-platform/DECISION_LOG.md`: 2026-05-01 BUGFIX + EVOLUTION entry.

## Architectural choice

Validation lives at `parse_state`, not at `Orchestrator._restore_phase_states`. Rationale captured in the DECISION_LOG entry: STATE.md is a human-readable, hand-editable contract per CLAUDE.md "On Every Commit"; a memory layer with that contract should fail fast at load time with a clear file-named error, not deep inside a downstream consumer with a stack trace that doesn't mention the file. Belt-and-braces at the orchestrator was considered and rejected — it would mask future malformed STATE.md without surfacing the file-level cause.

The other rejected alternative — making `phase_states` field type `dict[str, PhaseStatus]` in `_memory_models.py` so pydantic validates — would have coupled the memory model to orchestrator types. `_memory_models.py` is intentionally lower in the import graph than `_orchestrator_models.py`. The string-set + drift-guard test gets the same correctness without the coupling.

## Process learnings

- **A schema with closed values needs validation at the parser, not the consumer.** Same family as PR-005 (enforcement > aspiration) and PR-035 (hard gate enforcement vs aspirational contracts) — a contract that's only checked far from the boundary creates confusing error modes.
- **Memory layer files are hand-editable by design.** That's a feature, not a bug — humans and Claude both write to STATE.md across the session lifecycle. But every field with constrained values must surface a helpful error when an edit goes wrong, otherwise the contract leaks.
