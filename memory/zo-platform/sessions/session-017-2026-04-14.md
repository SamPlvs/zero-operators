## ZO Platform Learnings

### Learning 1: Denylist-first for DL data pipelines
When ZO's data pipeline agents build Phase 1 outputs, they should default to **denylist** (include all, exclude leakage) rather than **allowlist** (curated subset). Reasons:
- Curated tag lists from pre-project humans may be incomplete or biased
- Deep learning benefits from maximum feature space — let the model decide what's useful
- Feature selection is model-dependent (tree models use built-in; neural nets may use different approaches)
- The pipeline should exclude only what would cause target leakage, not pre-select features

**Implication for ZO:** Phase 1 subtasks should include a "leakage audit" step but NOT a "feature curation" step. Feature selection belongs in Phase 2 as a model-dependent transform.

### Learning 2: Specialist reviews catch real issues
Three specialist personas (domain, ML, data science) reviewing the same pipeline found complementary issues:
- Domain: missing flow transmitter exclusions (defence-in-depth)
- ML: trend slope units inconsistency, test coverage gaps
- Data science: detection limit handling, sample-to-feature ratios

**Implication for ZO:** The code-reviewer agent should invoke domain-specific review prompts when available. The data-scout should flag sample-to-feature ratios in Phase 1 reports.

### Learning 3: Notebook as executable validation
The Phase 1 notebook serves dual purpose: documentation AND validation. Executing all cells end-to-end catches API mismatches (wrong constructor args, renamed methods) that unit tests may miss if mocked.

**Implication for ZO:** Phase notebooks should be auto-executed as part of the gate check, not just generated as templates.
